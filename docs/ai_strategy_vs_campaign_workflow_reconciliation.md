# AI Strategy vs Campaign Workflow — Reconciliation

**Updated:** 2026-05-03

---

## Overview

This document defines the boundary between brand-level strategy and campaign-level strategy, and how they relate in the Marketing OS Lite AI architecture.

---

## Two Distinct Strategy Layers

| Layer | Scope | Persistence | AI Class | Route |
|---|---|---|---|---|
| Brand Strategy | Workspace-level, reusable across all campaigns | `brand_strategies` table | `OpenAIBrandStrategyProvider` / `MockBrandStrategyProvider` | `POST /api/brand-strategy/generate` |
| Campaign Strategy Adaptation | Campaign-specific, derived from brand strategy | `campaign_strategy_briefs` table | `OpenAIWorkflowProvider.generateStrategyBrief()` | `POST /api/campaign-workflow/strategy-brief` |

---

## Brand Strategy (Brand Level)

**What it is:**
A workspace-level, reusable strategic foundation built from the brand profile. Generated once (or updated occasionally) and shared across all campaigns.

**Input:**
- Brand profile fields: `brandName`, `productsServices`, `targetAudience`, `toneOfVoice`, `preferredChannels`, `forbiddenClaims`, `visualNotes`
- Optional: `userProvidedAnswers` (free-text context)

**Output:**
- `strategySummary` — 2-3 sentence overview
- `positioning` — brand's market position
- `idealCustomerProfile` — detailed ICP
- `primaryAudience` / `secondaryAudience`
- `keyMessages` — 3-5 core messages
- `valueProposition`
- `contentPillars` — 3-5 content themes
- `channelStrategy` — per-channel guidance
- `toneGuidelines`
- `ctaGuidelines`
- `forbiddenClaims` — safety guardrails
- `riskNotes`

**Lifecycle:** `draft` → `current` → `archived`  
**Only one `current` per workspace at any time.**

---

## Campaign Strategy Adaptation (Campaign Level)

**What it is:**
A campaign-specific adaptation of the brand strategy for one particular campaign. Generated per campaign inside the AI Workflow (Stage 2).

**Input:**
- Campaign intake: `name`, `objective`, `audience`, `product`, `channels`, `tone`, `offer`, `constraints`
- Brand profile fields (currently — to be updated to also pull from brand strategy, see below)

**Output:**
- `objective`, `targetAudience`, `positioning`, `keyMessage`, `recommendedChannels`, `contentAngles`, `ctaDirection`, `requiredAssets`, `missingContextWarnings`, `risksSafetyNotes`

**Lifecycle:** New row per generation, no archiving.

---

## Relationship

```
Brand Profile (user config)
       ↓
Brand Strategy Agent (workspace-level AI)
       ↓ saved to brand_strategies (status: current)
       ↓
Campaign Workflow Stage 2 (campaign AI — adaptation)
       ↓ saved to campaign_strategy_briefs
       ↓
Campaign Workflow Stage 3, 4, ...
```

---

## Current Wiring State (2026-05-03)

### What IS wired
- Brand profile → Brand Strategy Agent → `brand_strategies` table ✅
- Campaign workflow Stage 2 → `campaign_strategy_briefs` table ✅

### What is NOT yet wired (follow-up)
- Campaign workflow Stage 2 does not yet query `brand_strategies` as context

**Current Stage 2 behavior:**  
Uses brand profile fields directly (`brandName`, `toneOfVoice`, `targetAudience`, `forbiddenClaims`, `preferredChannels`) as inputs to `generateStrategyBrief()`.

**Target Stage 2 behavior (next slice):**  
1. After loading the campaign intake, also query `brand_strategies` for the workspace (`status = "current"`)
2. If found, pass `brandStrategy.strategySummary`, `brandStrategy.keyMessages`, `brandStrategy.positioning` as additional context into `StrategyBriefInput`
3. Update `OpenAIWorkflowProvider.generateStrategyBrief()` system prompt to reference brand strategy context

**Implementation effort:** Low — 1 DB read in `campaignWorkflow.ts`, 3 optional fields added to `StrategyBriefInput`, system prompt update.

## Governance Rules (Both Layers)

Neither the Brand Strategy Agent nor the Campaign Workflow AI may:
- Approve content
- Mark campaigns as ready
- Publish content
- Modify budget
- Upload assets
- Bypass role guards
- Bypass workspace scoping

All outputs are **draft-only** until a human reviewer approves and manually publishes.

## Source Tracking

Both layers track `source` on every generated row:

| Source | Meaning |
|---|---|
| `real` | Generated by OpenAI GPT-4o-mini with a real API key |
| `mock` | Generated by template-based mock provider (no AI call) |
| `unavailable` | AI key missing — draft returned but NOT saved |
